# Plan: Provider Management with Custom Provider Support

## Overview
Add a provider management UI where users can configure API keys, base URLs, and add custom providers with three API format options. Replace build-time env var injection with runtime-configurable provider configs stored via PreferencesService.

## New Types (`types/provider.ts`)

```ts
export type ApiFormat = 'chat_completions' | 'messages' | 'responses';

export interface ProviderConfig {
  id: string;                    // unique ID (e.g., 'gemini', 'custom-1')
  name: string;                  // display name
  apiKey: string;                // user-entered API key
  baseUrl: string;               // e.g., 'https://api.openai.com/v1'
  apiFormat: ApiFormat;          // which API format to use
  isEnabled: boolean;            // toggle
  isBuiltIn: boolean;            // true for default providers, false for custom
  models: string[];              // available model IDs (user can add)
  selectedModel: string;         // currently selected model
}
```

## New Files

### 1. `services/infrastructure/ProviderConfigService.ts`
- `loadProviderConfigs(): Promise<ProviderConfig[]>` — loads from PreferencesService, falls back to defaults
- `saveProviderConfigs(configs: ProviderConfig[]): Promise<void>` — persists
- `getDefaultConfigs(): ProviderConfig[]` — returns built-in providers with default base URLs (no API keys — user must enter them)
- `addCustomProvider(config): Promise<ProviderConfig[]>`
- `removeCustomProvider(id): Promise<ProviderConfig[]>`
- `updateProviderConfig(id, updates): Promise<ProviderConfig[]>`

### 2. `services/providers/GenericProviderService.ts`
A universal provider client that handles all 3 API formats:
- `createClient(config: ProviderConfig)` — returns an OpenAI SDK client for `chat_completions`, or a fetch-based caller for `messages`/`responses`
- `analyzeTradingView(config, prompt, images, ...)` — same signature as existing provider services
- `getQuickResponse(config, prompt, history, model)` — quick text response
- For `chat_completions`: use OpenAI SDK (existing pattern)
- For `messages` (Anthropic): direct fetch to `{baseUrl}/v1/messages` with Anthropic headers
- For `responses` (OpenAI Responses): direct fetch to `{baseUrl}/responses`

### 3. `components/settings/ProviderManager.tsx`
The main UI component:
- **Provider dropdown** — lists all configured providers (built-in + custom)
- **Selected provider config panel:**
  - API Key input (password field with show/hide toggle)
  - Base URL input (text field, pre-filled for built-in providers)
  - API Format dropdown: "Chat Completions (/v1/chat/completions)", "Messages (/v1/messages)", "Responses (/responses)"
  - Model selection dropdown (from provider's models list)
  - Enable/Disable toggle
  - "Test Connection" button (optional, sends a minimal request)
- **Custom provider section:**
  - "Add Custom Provider" button
  - Form: Provider Name, Base URL, API Key, API Format dropdown
  - Delete button for custom providers (not built-in ones)
- Styled with existing zinc dark theme + Tailwind classes

## Integration Strategy (Phased)

**Phase 1 (this implementation):** Build the UI + storage + GenericProviderService. Custom providers work immediately via GenericProviderService. Built-in providers still use their existing service files (backward compat).

**Phase 2 (future):** Migrate built-in providers to also use ProviderConfigService for API keys (replacing env vars). This makes all providers runtime-configurable.
