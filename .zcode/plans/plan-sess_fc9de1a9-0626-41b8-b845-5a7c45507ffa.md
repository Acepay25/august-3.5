# Plan: Task Component UI for Analysis "Thinking" Display

## Context
The spec's Task component design (collapsible step blocks, completion bar, composable wrapper) will be applied to the analysis progress display. Currently, progress is shown as a bottom overlay (spinner + status string) that replaces the input area. We'll replace this with rich collapsible Task blocks showing each pipeline step, plus a completion summary bar.

**Scope:** Sections 0-5 and 9 of the spec only. Sections 6 (Permission modal), 7 (Save Point removal), and 8 (Web search) are excluded — these concepts don't exist in this trading app.

## Step 3: Create `AnalysisProgress` Component

**New file: `components/AnalysisProgress.tsx`**

A composable wrapper panel (fulfills Section 4 + Section 9) that renders the analysis pipeline as Task blocks. This is the single wrapper that hosts all block types.

### Props (from App.tsx state):
```ts
interface AnalysisStep {
  id: string;
  title: string;           // "Fetching market data", "Running gate scan", etc.
  status: 'pending' | 'running' | 'complete' | 'error';
  subSteps?: AnalysisSubStep[];
  startTime?: number;
  endTime?: number;
}

interface AnalysisSubStep {
  label: string;
  detail?: string;
  filename?: string;  // rendered via TaskItemFile (basename only)
}

interface AnalysisProgressProps {
  steps: AnalysisStep[];
  isActive: boolean;       // true while analysis is running
  onCancel: () => void;
  isPostMortem?: boolean;  // purple theme variant
}
```

### Rendering:
- **While active (`isActive=true`):** Renders each step as a `Task` block:
  - `TaskTrigger`: status icon (spinner for running, checkmark for complete, error X for failed) + title + chevron. `defaultOpen=true` for the currently running step.
  - `TaskContent` → `TaskItem` per sub-step.
  - Cancel button at the bottom.
  - "Live View" / "View Post-Mortem" buttons preserved (existing functionality).
  
- **After completion (`isActive=false`):** Collapses into the **CompletionSummaryBar** (Section 9):
  - Chevron toggle + bold "Task completed"
  - Muted aggregate: e.g. "3 providers, 1 gate scan, 1 debate" (tallied from steps)
  - Elapsed time right-aligned (total duration)
  - Collapsed by default; expanding reveals the individual Task blocks

### Pipeline steps mapped from App.tsx `setLoadingMessage` calls:
| Step ID | Title | When |
|---------|-------|------|
| `market-data` | "Fetching market data" | Hybrid Intelligence fetch |
| `gate-scan` | "Running pattern gate scan" | GateKeeper |
| `analysis` | "Analyzing with {providers}" | Provider calls (sub-step per provider) |
| `debate` | "Ensemble debate" | Debate streaming (sub-step per round) |
| `validation` | "Validating outcome" | Post-mortem price validation |

## Step 4: Track Pipeline State in App.tsx

Add a new state array to track analysis steps:
```ts
const [analysisSteps, setAnalysisSteps] = useState<AnalysisStep[]>([]);
const [analysisStartTime, setAnalysisStartTime] = useState<number | null>(null);
```

Add a helper to update steps:
```ts
const updateStep = (id: string, update: Partial<AnalysisStep>) => {
  setAnalysisSteps(prev => prev.map(s => s.id === id ? { ...s, ...update } : s));
};
```

Wire into existing `setLoadingMessage` call sites — at each point where a status string is set, also update the corresponding step's status. The `loadingMessage` state is kept for backward compatibility but the overlay now renders `AnalysisProgress` instead of the simple spinner.
